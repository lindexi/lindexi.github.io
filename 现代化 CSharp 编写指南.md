## General Guidelines

- 网络层统一使用 `HttpClient` + `HttpRequestMessage`，避免 `HttpWebRequest`。
- JSON 处理优先使用 `System.Text.Json`，并尽量配合源生成上下文（`JsonSerializerContext`）提升 AOT 兼容性。
  - 将所有 `Newtonsoft.Json` 的功能替换为调用 `System.Text.Json` 实现
  - 禁止通过伪造兼容层保留 `Newtonsoft` 特性；应直接迁移为 `System.Text.Json` 特性与 API
- 对关键入参增加早期校验（如 `ArgumentNullException.ThrowIfNull` / `ArgumentException.ThrowIfNullOrWhiteSpace`），减少隐式空引用风险。
-. 常用集合初始化时尽量预估容量（如 `new List<T>(count)`、`new Dictionary<TKey, TValue>(capacity)`），减少扩容开销。
- 字符串拼接优先使用内插字符串（`$"..."`），替代 `string.Format`，可读性更高。
- 属性、方法、字段、类型名等的命名需要符合 C# 命名规范，例如：
  - 属性、方法、类型名应该使用 PascalCasing 命名风格
  - 局部变量应该使用 camelCase 命名风格
  - 字段应该使用 `_camelCase` 风格，而且尽可能使用 `readonly ` 标记只读。对于 `static readonly` 来说，应该视为常量，应该采用 PascalCasing 命名风格
- 最好不要公开字段，请封装为属性进行公开
- 考虑可空。但可以对进行网络请求的纯数据结构放宽要求
- 改造时，考虑 API 兼容性。旧 API 的命名保留兼容入口（`[Obsolete]`），新增规范命名 API 并逐步迁移

- 路径拼接应使用 `Path.Join` 方法，而非 `Path.Combine`
- 对于 UI 框架相关代码，需先理解整体代码实现逻辑，总结 MVVM 最佳实践，再按实践进行框架设计。界面代码与逻辑代码须严格分离，做到数据驱动。界面美化应抽取为独立的资源文件，对所有涉及的控件统一定义样式，并做好整体主题色调处理，同时确保设计时（Design-time）的良好支持

### 代码质量

- 所有公共 API 必须有 XML 文档注释
- Nullable reference types 启用，代码必须 null-safe
- 异步方法使用 `ConfigureAwait(false)`
- 优先使用 C# 最新语言特性（记录类型、模式匹配等）
- 遵循 KISS 原则，代码应简洁明了，避免过度设计
- 注释应说明"为什么"而非"是什么"，文件名和代码结构本身即文档
- 避免在代码中引用临时方案名称（如"方案A"、"Plan B"），这些会失去上下文
- 不保留仅包含注释的空文件，每个文件都应有实际代码价值

# 通用C#基础库/辅助类开发规范（AI生成通用提示词）
## 文档用途
用于指导AI生成**生产级、无业务耦合、可复用的C#基础类库/辅助类**，适用于任何业务场景（文件处理、音视频操作、网络请求、工具封装等），要求代码符合.NET生态规范、健壮性强、易维护、无耦合。

---
## 核心设计原则（所有场景通用）
1.  **实例优先原则**：优先实现为非静态实例类，仅无状态纯工具可使用静态类；禁止用静态类承载有状态逻辑
2.  **强类型优先原则**：所有入参/出参禁止用无意义的字符串、字典等弱类型传递，必须自定义强类型记录/类承载业务参数
3.  **可配置优先原则**：禁止硬编码配置项，所有可变参数支持上层传入，提供合理默认值
4.  **无副作用原则**：类库内部产生的临时资源、状态必须自动清理，无需上层调用者手动处理
5.  **无耦合原则**：禁止耦合特定业务逻辑、特定输出端（比如控制台、特定日志框架），支持上层自定义扩展

---
## 强制编码规范（所有场景必须遵守，禁止违反）
| 规则分类 | 要求 | 禁止行为 |
|----------|------|----------|
| 路径与IO处理 | 1. 所有文件/目录路径必须用`FileInfo`/`DirectoryInfo`强类型传递<br>2. 路径拼接必须使用`Path.Join`方法<br>3. 临时文件/目录名必须用`Path.GetRandomFileName`生成 | 1. 禁止用字符串传递路径参数<br>2. 禁止使用`Path.Combine`拼接路径<br>3. 禁止用Guid生成临时文件名<br>4. 禁止硬编码任何固定路径 |
| 异步与并发 | 1. 所有IO、进程、网络等耗时操作必须实现为全异步方法，方法名带`Async`后缀<br>2. 等待进程/IO/任务必须使用异步等待方法（比如`WaitForExitAsync`、`await`）<br>3. 所有异步方法必须支持`CancellationToken`取消令牌 | 1. 禁止用同步等待（比如`WaitForExit()`、`.Result`、`.Wait()`）阻塞线程<br>2. 禁止异步方法不支持取消操作 |
| 资源管理 | 1. 持有非托管资源、临时资源的类必须实现`IAsyncDisposable`接口<br>2. 类库内部产生的临时文件、连接、句柄必须自动清理，异常场景也不例外 | 1. 禁止让上层调用者手动清理类库内部产生的临时资源<br>2. 禁止资源泄漏 |
| 可观测性 | 1. 所有日志、错误信息必须通过通用回调/标准`ILogger`抽象透出<br>2. 异常必须保留完整堆栈，禁止吞异常，可包装后抛出 | 1. 禁止直接向控制台、文件等特定输出端写日志<br>2. 禁止隐藏错误信息，返回模糊的成功/失败状态 |
| 参数校验 | 所有公共方法入参必须做前置合法性校验，提前拦截无效输入 | 禁止不校验参数直接执行逻辑，导致内部抛出模糊异常 |

## API设计通用要求
1.  **无歧义入参**：禁止用`Dictionary`、`object`、字符串等弱类型承载结构化参数，必须自定义`record`/`class`承载业务参数，参数名要语义化
2.  **依赖注入友好**：依赖项（比如外部工具路径、配置、日志）优先通过构造函数传入，禁止在类内部硬初始化依赖
3.  **兼容性**：默认兼容.NET 6+ LTS版本，特殊版本要求需明确声明
4.  **可扩展性**：核心逻辑支持上层自定义扩展，禁止写死不可修改的逻辑

---
## 输出要求
1.  所有公共类型、方法必须包含完整的XML注释，说明用途、参数含义、返回值、异常情况
2.  必须包含上层调用示例代码，示例可直接运行
3.  代码必须符合.NET官方编码规范，命名遵循PascalCase/CamelCase标准
4.  异常要分类处理，业务异常和系统异常要明确区分
5.  无需业务场景特定的硬编码逻辑，可直接复用在不同项目中
